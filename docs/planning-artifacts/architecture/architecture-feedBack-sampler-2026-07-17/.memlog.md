- (decision) AD: ONE unified voice engine for all four formats; every format lowers at load time into the single SFZ-superset region/voice/mod-matrix model (extend the mod matrix with SF2 curve/source primitives). No per-format synthesis backends, ever - prevents bifurcated RT-safety/streaming/voice-stealing/author-API. Unified-engine invariant is primary; sfizz is the adopted foundation (seed), replaceable without violating it. FluidSynth stays test-oracle only, never a runtime dependency.
- (direction) User emphasis: unified engine is the invariant, not sfizz per se
- (decision) AD: One sample pool owns all sample memory and disk I/O policy (preload heads, stream tails past configurable RAM threshold; sfizz FilePool as adopted shape). Format frontends lower to regions holding references into the pool and never touch disk after lowering. SF3 Vorbis decoded to PCM at load/preload, never on audio thread; upstream of the pool SF3==SF2. Memory knobs = voice limit + pool budget only (NFR-7). Prevents: per-format sample handling, mmap page-fault RT violations, RAM-only design that can't retrofit streaming.
- (decision) AD: Three-tier state ownership. (1) Automatable control state owned by host-parameter layer (APVTS); UI and automation write ONLY through parameters; audio thread reads atomically; library-defined controls map onto a pre-allocated proxy-parameter pool (size TBD - spine number). (2) Structural state (loaded library/preset) mutated only via command path: background loader lowers to a complete immutable engine model, delivered to audio thread as atomic snapshot swap, retired off-thread; no incremental mutation of engine data. (3) Shared persistent config (folders, library index, settings) owned by config/library service in engine-core, per-user store, outside plugin state. Plugin state saves only library reference + sf-preset index + param values. Prevents: embedded/standalone divergence, DAW projects embedding library lists, RT-unsafe mutation.
- (decision) AD: Saved library reference is a resolvable identity (format + name + library-folder-relative path) with resolve-on-load fallback and explicit missing-library UI state - never a bare absolute path.
- (question) Open: proxy-parameter pool size for library-defined controls (VST3 cannot add params post-instantiation)
- (decision) AD: feedBack integrates the sampler EXCLUSIVELY as a standard VST3 loaded through its existing VST host - no bespoke in-process/static-lib embed path. One artifact for app and DAW users; host sandboxing gives fault isolation; FR-19 shared config already satisfied via per-user store; v1.5 author shape = feedBack plugin API driving the hosted VSTi. Retires the research addendum's 'core lib consumed by in-app engine' line. New dependency: feedBack's VST host must support VST3i + MIDI routing + editor embedding + session state - reality-check before binding.
- (question) Open: verify feedBack VST host capabilities (VST3 instrument hosting, editor window embed, state persistence) against sampler needs
- (decision) AD: sampler-core is a separate static-lib target; MAY use JUCE non-GUI modules internally (audio buffers/format readers, files, XML, threads); its PUBLIC API is JUCE-free (own model + std types only); zero UI/host/editor dependencies in core. Keeps a JUCE-free core reachable later by swapping internals behind a stable API. User delegated the call; criterion = best path to a performant, competitive sampler.
- (direction) User north star: quality/performance competitive with established samplers (Decent Sampler, Sforzando) - prefer pragmatic choices that maximize shipped quality over purity
- (decision) AD: Binding performance budget - 128 voices typical piano lib <=25% of one core @48kHz/256; fixed engine overhead <=150MB beyond sample pool; library switch command->audible <=2s preloaded. DSP inherits sfizz SIMD dispatch; no hand-rolled DSP.
- (decision) AD: Proxy-parameter pool fixed at 128; library controls map to first N proxies at load; beyond 128 controls remain UI-operable but not host-automatable. Oversized because VST3 forbids post-instantiation growth.
- (decision) AD: Global config = versioned JSON (settings.json + library-index.json) in platform per-user config dir; atomic temp-write+rename; schema-versioned, migrate-forward, fail-soft on newer schema; concurrent writers last-writer-wins. Owned solely by core's config service. SQLite swap behind same API if index scale demands - internal, not spine.
