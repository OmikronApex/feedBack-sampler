# corpus/ — the fidelity corpus (Story 1.6, AD-10)

Real instrument libraries rendered through the engine on every CI run and
diffed against checked-in reference captures. Fidelity is tracked
**per library, not per opcode**: an entry passes or fails as a whole, which is
what a user experiences. This directory is the seed of the Epic-7 scoreboard;
Epics 2 and 4 only extend the manifest (SF2/SF3, Decent Sampler).

## Layout

- `manifest.json` — schema-versioned entry list (see below). The single
  source of truth; CI cache keys hash this file.
- `midi/<id>.mid` — fixed MIDI sequence per entry (checked in; generated by
  `tools/make_midi.py`). The same bytes drive the external oracle capture and
  the offline runner.
- `reference/<id>.wav` — reference render per entry (checked in, PCM16
  stereo 48 kHz; quantization floor ~3e-5, orders of magnitude under the
  thresholds).
- `golden/<id>.expected.txt` — lowering snapshot per entry (Story-1.3
  serializer), so lowering drift and rendering drift are separately
  attributable.
- `cache/` — downloaded corpus assets + reports. Never committed
  (gitignored); CI restores it via `actions/cache`.
- `tools/run_corpus.py` — fetches pinned assets, runs
  `fbsampler-corpus-render` per entry, writes `corpus-report.json` +
  human summary. `tools/make_midi.py` regenerates the MIDI fixtures.

## Manifest schema (v2)

Per entry: `id`, `format` (`sfz` | `sf2` | `sf3`), `source` (repo + pinned
commit + raw base URL), `license` (SPDX or recorded license conclusion,
mandatory), the entry-point path (`sfz` for SFZ entries, `file` for
soundfont containers), soundfont entries additionally carry `bank` +
`program` (ONE preset rendered per entry), `midi` / `reference` / `golden`
(paths in this directory), `reference_provenance`, `render_frames`,
`rationale`, `files` (every fetched file with its sha256), optional legacy
`thresholds` overrides, optional `threshold_override` (numeric overrides +
mandatory `rationale` string — schema v2), and optional
`expected_fail: <issue ref>` (a tracked, diagnosed gap: the entry reports
FAIL but only the per-format PRD gate decides the job).

Adding an entry: pick a freely-licensed library (CC0 / CC-BY /
GPL-compatible — the `license` field is mandatory and must permit
redistribution or fetch-at-build), pin the source commit, list every file
with its sha256, add a fixed MIDI sequence in `tools/make_midi.py` covering
velocity layers, sustain and release tails, then run the capture procedure
below. Record the rationale for any threshold override in the manifest.

## Asset policy

Corpus assets are **fetched at build time**, never committed: the manifest
pins an exact commit and per-file sha256, `run_corpus.py` verifies every
download and hard-fails on mismatch. Keep entries small (≈1 MB each now);
references are checked in directly while they stay in the low-MB range —
if they outgrow that, move `reference/` to Git LFS (decision recorded here).

## Reference capture procedure

References are captured **once** from the external oracle and checked in —
CI never runs the oracle. For SFZ the oracle is **Sforzando** (closed-source,
GUI-only, not CI-able):

1. Load the entry's `.sfz` (exact pinned commit of the source repo) into
   Sforzando (record the exact Sforzando version in the commit message).
2. Host it in a DAW/offline-render host at **48 kHz**, stereo, no other
   plugins, all Sforzando controls at defaults.
3. Play the entry's `corpus/midi/<id>.mid` from time zero and bounce
   **offline** exactly `render_frames` samples (192000 = 4 s).
4. No normalization, no dither, no trimming: export as PCM16 stereo WAV to
   `reference/<id>.wav` and set the entry's `reference_provenance` to
   `oracle-sforzando-<version>`.

**Current status:** references are *provisional self-captures*
(`reference_provenance: provisional-self-capture`), recorded from this
engine via `run_corpus.py --update-references`. They pin today's behavior so
any rendering change is flagged from day one; swapping in true Sforzando
captures later only changes the baseline, not the machinery. FluidSynth
(Epic 2) can run headless, but the checked-in-capture pattern stays uniform.

Regenerating after an *intentional* engine change:

    python corpus/tools/run_corpus.py --tool <fbsampler-corpus-render> \
        --update-references --update-goldens

then inspect and commit the diff.

### FluidSynth capture procedure (SF2, Story 2.2)

For SoundFont behavior the oracle is **FluidSynth**, which runs headless, so
captures are scripted and re-runnable — but still captured once and checked
in (CI never runs FluidSynth). The Story-2.2 oracle fixtures live under
`tests/render/fixtures/sf2/` (kept out of the corpus until Story 2.5 folds
SF2 entries into the manifest):

1. FluidSynth **2.5.6** (Windows x64 cpp11 build; any platform's 2.5.x
   produces byte-comparable fast renders for these fixtures).
2. `fluidsynth -ni -g 1 -r 44100 -o synth.gain=1 -F <out.wav> oracle.sf2
   <fixture>.mid` — fast offline render, no realtime, chorus/reverb at
   defaults (the fixture routes nothing to effect sends).
3. No normalization, no trimming; PCM16 output as FluidSynth writes it.
4. `tests/render/fixtures/sf2/make_oracle_refs.py` performs 1–3 end to end;
   `tests/render/sf2_oracle_test.cpp` diffs behavioral metrics (normalized
   velocity curve, CC-sweep gain trajectory, f0 under pitch bend) against
   the captures with per-test thresholds and rationale in comments.

### FluidSynth capture procedure (corpus slice, Story 2.5)

The SF2/SF3 corpus entries carry **true oracle captures**
(`reference_provenance: fluidsynth-2.5.6`), produced by
`corpus/tools/capture_sf_references.py`:

- FluidSynth **2.5.6** (pin hard — minor versions change rendering)
- 48 kHz stereo, fast render, gain 1.0
- **reverb and chorus disabled** (`synth.reverb.active=no`,
  `synth.chorus.active=no`): our engine has no reverb/chorus, so diffing
  against effected output would measure the effects, not the sampler
- output trimmed/padded to the entry's exact `render_frames`, stored PCM16
  stereo (uniform with the Epic-1 reference format)
- the fixture MIDI embeds bank-select + program-change so FluidSynth plays
  the same preset the runner lowers by the manifest's `bank`/`program`
  (our engine ignores program changes — the preset is fixed at lowering);
  bank-128 entries ride MIDI channel 10

Threshold overrides for these entries are **calibrated to the measured
cross-engine diff** (schema v2 `threshold_override` with mandatory
`rationale`): sfizz and FluidSynth legitimately differ in interpolation,
envelope curve shape, and the engine's tracked 2.2 gaps (filter,
vibrato LFO). The thresholds pin the measured fidelity level so regressions
are visible, and tighten as gaps close. The PRD §6 gate (100% load /
>=90% render-pass per format) is asserted by `run_corpus.py` on every run.

**Weight decision (1.6 review deferral, resolved):** per-format percentages
remain unweighted — at the current corpus scale a `weight` field would have
no consumer; revisit when a format slice grows past ~10 entries.

**Repo size note:** the soundfont slice adds ~6 MB of references + ~4 MB of
lowering goldens (checked in). GeneralUser GS (~31 MB) and
MuseScore_General.sf3 (~40 MB) are fetch-at-build, cached by the existing
manifest-hash `actions/cache`; a cold CI run downloads ~70 MB once. If Epic
4's Decent Sampler slice pushes checked-in payload further, move references
to Git LFS then (single documented switch).

## Diff metrics and NFR-5 thresholds

Byte equality across engines/compilers is impossible; too-loose thresholds
measure nothing. Defaults (in `manifest.json`, per-entry overrides allowed
with a recorded rationale):

- `peak` (1e-2): max absolute per-sample deviation — catches a wrong sample,
  missing velocity layer or broken loop by orders of magnitude.
- `rms` (1e-3): whole-render RMS deviation — catches broad gain/envelope
  drift.
- `window_rms` (3e-3, 4096-frame windows): worst local RMS deviation —
  catches short-lived glitches a whole-render RMS would average away.
- Duration/shape check: the render must be exactly `render_frames` stereo
  frames and match the reference's shape.
- Silence guard: total energy must clear a floor, so a dead engine can never
  pass a silent-vs-silent comparison.

The report (`corpus-report.json`) carries per-entry metrics plus per-format
`load_pct` / `render_pass_pct` — the PRD §6 v1 gate (100% loads, ≥90%
renders correctly per format) reads straight off those two numbers.

## CI and scaling

The `corpus` CI job (Linux) builds `fbsampler-corpus-render`, restores
`corpus/cache` keyed on the manifest hash, runs the suite on every push, and
uploads `corpus-report.json` as an artifact; any entry failure fails the job
with the per-library table in the log. The corpus is small (3 entries,
~3 MB), so it runs per-push on one platform. When it grows: keep a small
representative subset per-push and move the full corpus to a nightly job —
the manifest gains a `tier` field at that point, nothing else changes.
